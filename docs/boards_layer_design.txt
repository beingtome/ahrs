1. 一句话定位
================================================================================

Boards 层回答唯一一个问题：这块板子上，什么东西接在哪个总线的哪个地址上？

它不是驱动，不是 HAL，不是业务逻辑。
它就是一张编译时固化的"硬件接线表"。


================================================================================
2. 架构位置与协作关系
================================================================================

  L5  应用层        零硬件依赖，只通过"设备名"发现和操作传感器
  L4  中间件层      零硬件依赖，只消费算法输入输出
  L3  传感器驱动层  依赖 L2 的总线抽象接口，不感知芯片型号
  L2  驱动抽象层    定义跨平台的类型系统（总线接口、GPIO 描述、设备注册）
  L1  芯片支持层    实现 L2 接口，把厂商 HAL 翻译成统一抽象
  L0  BSP 层       ← 本层

协作关系（谁依赖谁、谁给谁什么）：

  boards → chips:
    boards 调用 chips 的工厂函数创建总线实例。
    boards 告诉 chips"初始化这个引脚"。
    boards 不知道 chips 内部怎么做的，只知道它能做。

  boards → drivers/hal (L2):
    boards 调用 L2 的设备注册 API，把板载设备信息注入全局设备表。
    boards 使用 L2 定义的类型（GPIO 描述、设备类型枚举、总线类型枚举）。

  上层 → boards:
    不允许。L1~L5 禁止 include boards 层任何文件。
    上层通过 L2 的全局设备表间接获取 boards 层注册的资源。
    换板子时，上层代码纹丝不动。


================================================================================
3. 一个"板子"本质上是什么
================================================================================

一块板子 = 一个具体的物理硬件，它承载了：

  a) 一颗主控芯片（MCU）
  b) 时钟源（外部晶振或内部 RC）
  c) 供电系统（电压域、LDO、电池管理）
  d) 调试接口（SWD/JTAG）
  e) 若干板载外设（传感器、LED、按键、存储等）
  f) Flash/RAM 分区方案（Bootloader 区域、APP 区域、参数存储区域）
  g) 启动配置（BOOT0/BOOT1 引脚状态）

同一个 MCU 型号，装在不同板子上就是不同的 board。
两块板子可能 MCU 相同，但外设完全不同，这就是 board 层存在的意义。


================================================================================
4. 硬件资源分类
================================================================================

4.1 任何板子都必须有的资源（必选）
────────────────────────────────

  a) 主控芯片：
     任何板子有且仅有一颗 MCU。它的型号决定了可用外设的上限。
     但 boards 层不直接操作 MCU 寄存器——那是 chips 层的工作。

  b) 电源系统：
     供电路径、电压等级。决定了板子能在什么电压范围工作。
     boards 层至少需要声明供电电压（影响 ADC 参考电压等）。

  c) 时钟系统：
     外部晶振频率、PLL 配置决定了系统主频。
     时钟配置的代码由 CubeMX（或其他工具）生成，但配置参数本身是板级决策。
     "用 8MHz 还是 25MHz 晶振"是 board 的问题，不是 chip 的问题。

  d) 调试接口：
     SWD 或 JTAG，引脚通常是固定的（如 PA13/PA14），但某些板子可能复用。

4.2 板载交互类（人机接口，常见但非必须）
──────────────────────────────────────

  - LED：指示灯，通常 GPIO 推挽输出，数量 0~N
  - 按键：用户输入，通常 GPIO 输入 + 中断或轮询
  - 蜂鸣器：PWM 驱动或 GPIO 高低电平
  - 显示屏：SPI/I2C/8080 接口的 LCD/OLED

4.3 传感器类（AHRS 项目核心，常见但非必须）
────────────────────────────────────────

  - IMU（惯性测量单元）：加速度计 + 陀螺仪，I2C 或 SPI 接口，通常带中断引脚
  - 磁力计：三轴地磁传感器，I2C 或 SPI 接口
  - 气压计：气压/高度传感器，I2C 或 SPI 接口
  - GPS/GNSS：定位模块，UART 接口
  - 测距仪：激光/超声波测距，UART 或 I2C 接口
  - 光流传感器：视觉运动传感器，SPI 接口

  对 boards 层来说，所有这些传感器的角色完全一样：
    一个名字 + 一条总线 + 一个地址 + 若干引脚（中断/数据就绪/片选）。

  一个传感器驱动只用"名字"查到自己连在哪条总线上即可工作。

4.4 通信接口类（连接外部设备或主机）
────────────────────────────────

  - USB：设备模式（虚拟串口、HID）或主机模式
  - CAN/CAN-FD：工业总线，需要收发器芯片
  - RS485/RS232：工业串口，需要电平转换芯片
  - 以太网：需要 PHY 芯片
  - 无线模块：WiFi/BLE/Zigbee，通常通过 UART 或 SPI 连接

  这些接口在 boards 层的角色：
    声明总线实例并将其暴露给上层。上层不关心是 STM32 的 USART2 还是 ESP32 的 UART_NUM_1，
    只看到一个"串口"。

4.5 存储类
──────────

  - 外部 Flash（SPI NOR/NAND）：存储固件、日志、参数
  - 外部 SRAM/PSRAM：扩展内存
  - MicroSD 卡：SPI 或 SDIO 接口
  - EEPROM：I2C 接口，存储校准参数

4.6 配置参数类（不是硬件，但随板子不同而变化）
────────────────────────────────────────────

  - 系统时钟频率
  - 各总线速率（I2C Fast/Standard, SPI 时钟频率）
  - 供电电压
  - 功能开关（是否启用 FPU、I-Cache、D-Cache）
  - 板级特性标记（BOARD_HAS_IMU = 1/0）
    → 这些宏通过构建系统传给编译器，上层可做条件编译但不 include boards 头文件


================================================================================
5. 文件结构设计
================================================================================

boards/<board_name>/
├── board.h             拓扑声明：板上有什么设备、接在哪个引脚上
├── board.c             初始化：按拓扑描述执行硬件初始化 + 注册设备
├── board_config.h       配置参数：时钟频率、供电电压、功能开关
├── cubemx/             CubeMX 生成代码（仅 STM32，不直接编辑）
├── linker/             链接脚本（Flash/RAM 分区是板级决策）
└── startup/            启动文件（中断向量表、堆栈初始化）

一个板子目录是全自包含的：换板子 = 换整个 boards/<name>/ 目录 + 改构建目标。

三个文件的职责严格分离：
  board.h       = "有什么、接在哪"（拓扑）
  board_config.h = "跑多快、怎么供电"（配置）
  board.c       = "怎么初始化"（过程）


================================================================================
6. board.h — 拓扑声明
================================================================================

只声明不执行。暴露给外界的信息只有：

  a) 板子名字和 MCU 型号
  b) 每个引脚的完整描述（端口 + 引脚号 + 模式 + 上下拉）
  c) 每个板载设备的完整描述（名字 + 类型 + 总线类型 + 地址 + 关联引脚）
  d) 全局总线实例的声明（供传感器驱动获取）
  e) 三个公开 API 的声明：初始化板子、获取设备表、获取板子名字

"引脚的完整描述"是关键设计决策：
  不用裸数字宏（如 #define IMU_INT_PIN GPIO_PIN_8），
  而用一个包含 port、pin、mode、pull 的复合描述。
  这样 board.c 中可以用统一的循环初始化所有引脚，
  芯片层也收到了完整的初始化参数，无需额外查表。

"板载设备描述"是 boards 层内部自用的数据结构。
  其他层不知道也不关心这个结构体的存在。
  board.c 遍历它，调用 L2 的 device_register() 把信息翻译到全局设备表中。
  之后上层通过 L2 的 device_find("icm42688") 获取连接信息。


================================================================================
7. board_config.h — 配置参数
================================================================================

只存储板级配置常量。"为什么某些配置放在 board_config.h 而非 board.h？"

  board.h 的配置 = 改变了意味着"硬件接线变了"（引脚不同、设备不同）
  board_config.h 的配置 = 改变了意味着"同硬件但参数不同"（降频运行、调整波特率）

典型内容：
  - 时钟频率（HSE 频率、PLL 倍频系数、系统主频）
  - 总线速率（I2C 100k/400k/1M, SPI 1M/10M/20M）
  - 供电电压（影响 ADC 参考和传感器供电判断）
  - 调试串口选择（USART1 还是 USART2，115200 还是 921600）
  - 硬件使能开关（是否启用 FPU/DCache/ICache）
  - 板载特性标记（BOARD_HAS_IMU 等宏）

BOARD_HAS_xxx 的使用策略：
  这些宏通过构建系统（-D 参数）传给编译器，上层可以用 #if 条件编译。
  但上层不 include boards/ 的任何文件，只间接使用这些宏。
  boards 层也通过设备表的存在性隐式表达"有没有 IMU"：
    如果 board_devices[] 里有 icm42688 → 有 IMU
    如果 board_devices[] 里没有 → 没有 IMU
  两种方式可以共存，互相校验。

8. board.c — 初始化流程
================================================================================

board.c 是 boards 层唯一同时引用 board.h 和芯片 HAL 的文件。
其他文件（board.h、board_config.h）禁止引入芯片厂商头文件。

初始化四步，顺序不可调换：

  步骤 1：芯片层基础初始化
    芯片系统时钟配置 + 芯片厂商 HAL 库初始化。
    boards 层不自己配置时钟寄存器，只调用 chips 层的 chip_system_init()。
    chips 层内部会调用 CubeMX 生成的 SystemClock_Config()。

  步骤 2：板级设备引脚初始化
    先做 GPIO 全局使能（CubeMX 的 MX_GPIO_Init），
    然后遍历设备表，初始化每个设备关联的 INT、DRDY、CS 引脚。
    LED 引脚也在这一步初始化。

  步骤 3：创建总线实例
    调用 CubeMX 生成的外设初始化函数（MX_I2C1_Init、MX_SPI1_Init 等），
    然后调用 chips 层的工厂函数，把 CubeMX 的 HAL 句柄包装为 L2 的 bus_i2c_t / bus_spi_t。

    注意区别：
      外设初始化（MX_I2C1_Init）是 CubeMX 生成的，做的是：配置引脚复用 + 时钟 + 外设参数
      工厂函数（stm32_i2c_create）是 chips 层提供的，做的是：把 HAL 句柄包装为 L2 抽象类型

 步骤 4：注册设备到全局设备表
    遍历设备表，根据每个设备的总线类型（I2C/SPI）和地址，
    调用 L2 的 device_register() 把设备信息写入全局表。
    之后 L5 应用层就可以通过 device_find("icm42688") 找到设备。

一个重要的"不做"：
  board.c 不调用任何传感器驱动的 create 函数。
  board.c 不知道板子上焊的是什么型号的 IMU——它只知道"这里有一个 I2C 地址为 0x68 的设备"。
  实例化具体传感器驱动是 L5 应用层的事。
  换传感器型号 = 只改 L5 应用层，board.c 不变。


================================================================================
9. CubeMX 协作策略
================================================================================

CubeMX 是 STM32 平台的代码生成工具。它生成的代码与我们手写的 boards 代码的关系：

  a) CubeMX 的职责：
     - 引脚复用配置（哪个引脚用 AF5 做 SPI1_SCK）
     - 外设时钟配置
     - 外设底层初始化函数生成（MX_I2C1_Init、MX_SPI1_Init）
     - HAL 句柄定义（hi2c1, hspi1 等）

  b) 我们的 boards 代码的职责：
     - 声明板上有什么设备（设备表）
     - 调用 CubeMX 生成的初始化函数
     - 将 CubeMX 的 HAL 句柄传给 chips 层的工厂函数
     - 注册设备到 L2 全局表

  c) CubeMX 重新生成时的隔离策略：
     CubeMX 生成的文件放在 boards/<name>/cubemx/ 目录下。
     board.c / board.h / board_config.h 不在 cubemx/ 下，不会被覆盖。
     CubeMX 重生成时只更新 cubemx/ 目录内容，我们的代码毫发无损。

  d) CubeMX main.c 的 main() 冲突处理：
     CubeMX 生成的 main.c 包含它自己的 main() 函数，与我们的 L5 应用层入口冲突。
     解决方案：
       - 我们的 L5 main() 是整个系统的唯一入口
       - CubeMX main.c 参与编译（因为 SystemClock_Config() 在里面）
       - 但 CubeMX 的 main() 通过宏或条件编译被排除
       - 或者 CubeMX main.c 不参与编译，需要的函数通过 extern 引用

10. 关键设计决策
================================================================================

决策 1：设备表用编译时常量数组，不用运行时动态构建
  板上有什么传感器在编译时就确定了，不需要运行时改。
  常量数组放在 .rodata 段，零 RAM 开销，零运行时开销。

决策 2：board_init() 只注册资源，不实例化驱动
  如果 board.c 调用了 icm42688_create()，换传感器就要改 board.c。
  正确做法：board.c 只声明"此处有一个设备"，L5 应用层负责实例化驱动。
  换传感器只改 L5 应用层。

决策 3：引脚描述用复合字面量，不用分散的宏
  裸宏（#define IMU_INT_PIN GPIO_PIN_8）只有 pin 号，缺少 port/mode/pull。
  复合字面量包含完整信息，board.c 可以用一个循环统一初始化所有引脚。

决策 4：对外仅暴露 3 个 API
  board_init()、board_get_devices()、board_get_name()。
  暴露越少，上层耦合越浅，换板子越容易。

决策 5：链接脚本归 boards/ 管，不归 chips/ 管
  Flash 分区（APP 起始地址、Bootloader 占用空间、参数存储区）是板级决策。
  同一芯片的不同板子可以有不同的 Flash 分区方案。

决策 6：boards 层是唯一知道"芯片型号"的层
  board.h 中有 BOARD_MCU 宏声明芯片型号。
  board.c 中根据芯片型号调用对应的工厂函数名。
  L1~L5 的代码不包含芯片型号相关的条件编译。


================================================================================
11. 添加新板子的流程
================================================================================

  步骤 1：创建 boards/<board_name>/ 目录
  步骤 2：board.h — 写引脚定义 + 设备表声明（板上有哪些设备、接在哪个总线/地址）
  步骤 3：board_config.h — 写时钟频率、总线速率、供电电压、功能开关
  步骤 4：board.c — 写初始化四步（调用芯片工厂 + 注册设备）
  步骤 5：放入 linker/ 和 startup/
  步骤 6：构建系统注册新板子

  影响范围：仅 boards/ 和构建配置。
  无需修改：chips/、drivers/、middleware/、app/ 的任何代码。

  注意事项：
    - 如果用的是已有的芯片平台（如 STM32G4），chips/stm32/g4xx/ 完全不需要改
    - 如果用的是新芯片平台（如从 STM32 换到 AT32），需要先在 chips/at32/ 下实现芯片层适配
    - 不管怎么换，drivers/sensors/ 和 app/ 都不需要改


================================================================================
12. 跨平台兼容性
================================================================================

boards 层的接口设计（设备表格式、初始化流程）是平台无关的。
差异仅体现在 board.c 内部调用的工厂函数名和传入的 HAL 参数类型：

  STM32：创建总线时传入 CubeMX 生成的 HAL 句柄指针
  AT32：传入 Artery 的 HAL 句柄（概念相似，类型名不同）
  ESP32：传入整数端口号 + GPIO 引脚号（因为没有 HAL 句柄这个概念）

但三种情况返回的都是同一个 bus_i2c_t*，上层完全无感知。

关键设计原则：
  不试图在 board.c 中用 #ifdef 支持多平台。
  每个平台的 board.c 是独立的——换平台 = 换 board.c，但 board.h 的声明格式不变。

附录：术语
================================================================================

  BSP            Board Support Package，板级支持包（= boards/ 目录）
  设备表         board_devices[]，板载硬件拓扑的编译时常量数组
  总线实例       bus_i2c_t* / bus_spi_t*，L2 层定义、L1 层创建、boards 层持有的运行时指针
  工厂函数       芯片层的创建函数，把厂商 HAL 句柄包装为 L2 抽象总线实例
  依赖注入       总线实例在 board_init() 中创建，通过设备注册传入全局设备表
  全局设备表     运行时设备注册表（L2 层维护），boards 层往里写，应用层往外读
  复合字面量     C99 描述的引脚配置（port + pin + mode + pull 四元组）
